# Evaluation Plan

## Research Question

At equal or explicitly measured cost, when does a heterogeneous inexpensive panel outperform a strong inexpensive single model, self-review, or repeated sampling?

## MVP Baselines

The first runner includes:

| ID  | Configuration                                                        |
| --- | -------------------------------------------------------------------- |
| A   | Direct inexpensive single model, one call                            |
| B   | Same model plus one self-review/repair call                          |
| C   | Same model, two independent samples plus inexpensive aggregation     |
| D   | Two heterogeneous inexpensive candidates plus inexpensive aggregator |

Future baselines:

- Optional third candidate on disagreement.
- OpenRouter managed Fusion as external reference. Current OpenRouter docs describe managed Fusion as a panel/judge pipeline with web search/fetch available to the inner calls, so compare it as a separate reference condition unless retrieval budgets are matched across all baselines.
- Retrieval-enabled or managed-router references should use a separate runner/report schema. The current fixed MVP matrix is for ordinary chat-completion model IDs with token-price snapshots, not hidden tool loops or managed inner deliberation.
- Router, latest-resolution, model-variant, plugin-enabled, or retrieval-enabled runs are external-reference conditions unless they get their own runner/report schema and matched evidence/cost budget.

## Report Artifacts

- Private reports retain the model, provider, price, usage, trace, prompt-hash metadata, optional run-provenance fingerprints, and optional self-reported normalized CLI invocation metadata needed for audit orientation and reproduction.
- Public reports are separate allowlisted projections, not redacted private reports.
- Public reports omit raw case IDs, run IDs, case-set hashes, prompts, answers, traces, model/provider identifiers, price snapshots, usage rows, raw failure details, per-case cost/latency, config/model provenance digests, paths, normalized invocation details, command lines, and environment details.
- Public reports still disclose generated row labels, per-case pass/fail outcomes, grader check kinds, grader evidence tier counts, configuration IDs, failure status categories, and evaluation-shape metadata. If the evaluated case file is public and ordered, public row-level outcomes can be linked back to that file.
- Public reports disclose only a no-digest case-manifest binding status from the private report: whether a manifest was bound, its intended use, hash algorithm family, privacy class, and schema/fingerprint versions. They omit canonical digests, raw-file digests, row hashes, manifest paths, and HMAC key metadata.
- Public reports disclose only no-digest case-set claim-gate status from the private report: pass or blocked status plus non-sensitive target, scope, and overall external-evidence-required status. They omit detailed case-set blockers, category and difficulty evidence, duplicate-content evidence, manifest digests, row hashes, and raw labels.
- Public reports disclose only no-digest run-provenance shape metadata from the private report: whether resolved-config, effective model-price fingerprint, fixed OpenRouter request-policy, and provider endpoint routing fields are present, the schema/content labels, the evaluated-config count used for public consistency checks, and the closed provider endpoint pinning status. The private report stores provider routing evidence such as `allowFallbacks`, ordered provider slugs, order count, and full-endpoint count. Public reports omit those slugs/details, config/model digests, config identities, model IDs, prices, source paths, normalized invocation details, commands, and environment details.
- Aggregation ledgers are private result metadata. They are retained only when output retention is enabled, are string-truncated like retained answers, and are not projected into public reports or public claim gates.
- Public reports expose top-level grader evidence tier counts over all cases plus separate scored-case tier counts, dominant scored tier, profile mix, and small-cell warnings. These fields remain available even when category breakdowns are suppressed.
- Public category breakdowns are descriptive stratification, not named-domain claims. They use generated category labels, disclose that labels are pseudonymous rather than anonymous, suppress the entire breakdown if any scored category has fewer than five distinct scored cases, include category-level grader evidence tier counts, warn when nonzero tier cells are smaller than five cases, and recommend at least 30 scored cases per category before category-level claims.
- Public reports label reproducibility as `private-audit-only`; the private report is required for exact model/provider and price provenance.
- Public aggregate cost/latency is rounded and suppressed below the small-sample warning threshold.
- Public report schema `frugal-fusion-public-eval-v11` includes `claimGate`, a machine-readable evidence gate over public projection fields plus no-digest private-report attestations. It can block unsupported top-level public report schema versions, malformed public disclosure contracts, unknown/duplicate/extra public config IDs, missing fixed MVP matrix configurations (`direct`, `self_review`, `repeated`, and `fusion`), unknown case-manifest or run-provenance schema/fingerprint/canonicalization/content labels, non-exact no-digest disclosure shapes for the disclosure root, case-manifest binding, case-set claim gate, run provenance, OpenRouter request policy, and provider routing, no-digest run-provenance evaluated-config counts that do not match the public report and fixed matrix, missing no-digest OpenRouter request-policy provenance, missing or non-exact provider endpoint pinning, malformed confidence-interval metadata, missing scored attempts or incomplete scored-attempt coverage for required configurations, missing task pass-rate intervals, missing cost-per-pass intervals, incomplete direct-vs-fusion pairing, missing direct-vs-fusion delta intervals, missing no-digest run-provenance disclosure, missing or blocked no-digest case-set claim-gate disclosure, underpowered category evidence, scored evidence with fewer than five non-surface grader-evidence cases overall, and claim-eligible categories with fewer than five non-surface grader-evidence cases. Public report claim gates require each required configuration to have one scored attempt for every scored case-trial, confidence intervals to be labeled as case-cluster bootstrap at the 95% level with at least 500 resamples, and private run provenance to report `single_provider_endpoint_pinned`. For manifest-bound `eval --public-out` runs, the CLI now fails before model calls, model-snapshot loading, or output writes when the no-spend case-set gate is blocked, exact provider endpoint pinning is absent from the static config, or `--bootstrap-samples` is below 500. Published category rows include within-category task pass-rate and pass-rate-delta intervals computed with the same case-cluster bootstrap contract, while category-level cost intervals remain omitted. Public v11 also exposes `choice_valid` and `choice_expected` check kinds for strict closed-label graders plus no-digest OpenRouter request-policy and provider endpoint routing disclosures; `choice_valid` is format validity, not partial task credit. Category delta intervals are descriptive robustness evidence and are not part of `claimGate.directionalComparison`, which remains based on the top-level direct-vs-fusion interval. The schema-version, disclosure-contract, public-config-list, confidence-interval metadata, evaluated-config count, OpenRouter request-policy status, and provider endpoint pinning status checks are public-shape consistency checks only; they do not parse arbitrary untrusted JSON or prove that no private report ever existed with richer fields. Config identity, transport-policy custody, provider endpoint custody, and account/BYOK routing settings still require private reproduction artifacts. Difficulty, category-by-difficulty coverage, and non-surface grader-evidence coverage by difficulty are enforced by the no-spend case-set gate and surfaced publicly only as pass/blocked attestation. A public report gate passing is insufficient without matching private reproduction artifacts and external holdout-process evidence. The public-report `minimums` object exposes the public-side non-surface evidence floors; detailed difficulty floors live in the private no-spend gate output. `overallClaimStatus` remains `external_evidence_required`.
- `verify-public-report` is a no-spend public artifact verifier for saved public reports. It reads only the public JSON, recomputes the current claim gate from public `schemaVersion`, `configs`, `disclosure`, and `metrics`, checks exact top-level and nested allowlisted public report fields, confirms public row-level outcomes support the aggregate case, scored-attempt, pass-rate, direct-vs-fusion pairing, and fusion-harm metrics, and confirms the embedded `claimGate` matches the recomputed gate. It exits 0 only when the public artifact shape is current, public cases match the published aggregate metrics, the embedded gate is consistent, and the recomputed gate has no blockers; stale, malformed, blocked, or mismatched reports exit 2 with public-safe JSON. It does not read private reports, manifests, model snapshots, configs, case files, APIs, or provider accounts, and it does not prove holdout isolation, private artifact custody, model/provider provenance, price provenance, or account/BYOK routing settings.

## Case Sets

- `examples/cases.jsonl` is a tiny smoke/development example.
- `examples/cases.public.jsonl` is a larger public harness sample with 54 scored cases, equal-count across nine generic categories, including a synthetic `citation_mechanics` category for bracket citation discipline plus strict `choice`, JSON, and numeric cases for deterministic non-surface grader plumbing. Each category has author-supplied closed-enum difficulty coverage metadata with two `easy`, two `medium`, and two `hard` cases, at least three non-surface grader-evidence cases, and at least one non-surface grader-evidence case in each difficulty bucket. It is useful for exercising the harness and grader mix, but it is not a locked benchmark or holdout; citation checks are mechanical source-id discipline rather than semantic entailment, choice checks are closed-label exactness rather than semantic classification quality, JSON/numeric checks are mechanical format or arithmetic checks, and the difficulty labels are coverage metadata rather than calibrated benchmark difficulty.
- `examples/cases.public.manifest.json` is a separate explicit artifact for the public sample. It uses schema `frugal-fusion-case-set-manifest-v4` and contains raw-file and canonical case-set fingerprints, row hashes, grader-kind counts, grader evidence tiers, aggregate difficulty coverage, equal-count category evidence, and claim-readiness warnings. It is intentionally not embedded in public evaluation reports.
- Manifests are linkability artifacts. Publish them only for intentionally public or frozen case sets; for private or unreleased holdouts, keep manifests private unless a salted/private-audit fingerprint scheme is added.
- Manifest row hashes are unsalted and include grader values and difficulty metadata, including choice labels, JSON required paths/equality/includes terms/array-position expectations, citation source IDs/claim snippets, and schema-subset literals such as enum values and property names, in the canonical row identity. This is acceptable for intentionally public samples, but private or unreleased holdout manifests can leak linkability against guessable prompts or answer keys.
- Private-audit HMAC manifests can be generated with `--manifest-hmac-key-env`. They use keyed `rawFileHmacSha256`, `canonicalHmacSha256`, `canonicalRowHmacSha256`, and `rawLineHmacSha256` fields instead of raw SHA-256 fields. They are reproducible only by auditors with the same secret and are not public reproducibility proof.
- HMAC manifests still disclose structural metadata such as row count, line numbers, row order, generated row IDs, smoke flags, grader families, grader evidence tiers, aggregate difficulty coverage, and aggregate balance. The CLI rejects source labels, public case IDs, public category labels, and `--private` summaries in HMAC mode.
- Evaluations can be bound to a manifest with `eval --case-manifest`. Binding verifies the manifest's canonical case-set digest before any model calls. HMAC manifests require `--case-manifest-hmac-key-env`; SHA-256 manifests may bind development or intentionally public sample runs, but `holdout` evaluation binding requires HMAC. For HMAC-bound reports, the private report stores the verified HMAC binding and omits the unkeyed `caseSetHash`.
- The private report `caseSetHash` uses the same difficulty-aware canonical case content as manifest v4 under the `case-set-canonical-v3` domain, but it remains private audit metadata and is omitted for HMAC-bound holdouts.
- Case files can be checked locally with `validate-cases` before any model spend. The default CLI summary omits raw category labels; `--private` is for local taxonomy review only. Use `--manifest-out` only when intentionally creating a separate case-set identity artifact, and use `--source-label`, `--public-category-labels`, and `--public-case-ids` only when those labels are safe to disclose.
- `validate-cases --claim-gate public_cost_performance` is a no-spend case-set gate for CI. When validation and assessment run successfully, it emits public JSON with `claimGate`; it exits 2 and skips manifest output when case-set blockers exist. It rejects `--private`, source labels, public case IDs, and public category labels in claim-gate mode. Parse and validation failures are collapsed to a generic public blocker so CI logs do not expose private row identifiers or grader values. Private holdout claim gates require HMAC manifests; unsalted SHA-256 manifests are linkability artifacts for intentionally public or frozen case sets and do not pass private holdout claim gates. The gate blocks exact duplicate scored case content using an internal task/constraints/smoke/grader fingerprint that ignores case IDs, category labels, and difficulty labels; only aggregate duplicate counts are exposed. It also blocks high-confidence lexical near duplicates using deterministic token-set similarity over scored task and constraint text only; smoke-only cases are excluded, grader values are not used for this near-duplicate check, and only aggregate counts plus thresholds are exposed. The gate also requires every scored case to have closed-enum difficulty metadata (`easy`, `medium`, or `hard`), at least 30 scored cases in each difficulty bucket, at least five scored cases in each difficulty bucket within every claim-eligible category, and at least five non-surface grader-evidence scored cases in each difficulty bucket. These are coverage floors, not proof of semantic difficulty calibration, and the duplicate checks detect literal or conservative lexical scored-content reuse, not paraphrases, semantic near-duplicates, answer-key reuse, topical clustering, or tuning contamination. The gate still requires external evidence even when `status` is `case_set_constraints_met`.
- Public cost-performance claims require a frozen and archived case manifest bound to the private report, a holdout split that was not used while tuning prompts/models/graders, the exact fixed MVP matrix (`direct`, `self_review`, `repeated`, and `fusion`) on the same cases, case-cluster bootstrap confidence intervals at the 95% level with at least 500 resamples, model/provider and price provenance, and a clear distinction between public sample results and private-audit reproduction data. The public-report claim gate blocks malformed public disclosure contracts, malformed public config lists, malformed confidence-interval metadata, missing manifest binding, non-holdout binding, holdout bindings that are not HMAC/private-audit, missing or blocked no-digest case-set claim-gate attestation, missing or count-mismatched no-digest run-provenance disclosure, missing fixed-matrix configs, and missing metric or cost interval cells for required configs. `eval --public-out --case-manifest` computes the no-spend case-set gate after manifest verification and also checks static public-report prerequisites that are knowable from CLI/config inputs; a blocked case-set gate, absent exact provider endpoint pinning, or fewer than 500 bootstrap resamples fails before model calls, model-snapshot loading, runtime construction, or output writes. Passing the case-set and public-report claim gates is not claim approval; generating or binding a manifest locally is not by itself public provenance.

## Fairness Controls

- Same case set and input for each configuration.
- Fixed-baseline configuration roles must use concrete model IDs. Managed OpenRouter router slugs such as `openrouter/fusion`, `openrouter/auto`, `openrouter/free`, `openrouter/pareto-code`, and `openrouter/bodybuilder`, latest-resolution aliases beginning with `~`, and model variant suffixes such as `:online`, `:nitro`, and `:exacto` are external-reference conditions rather than direct/self-review/repeated/fusion matrix cells.
- OpenRouter fixed-baseline calls request router metadata and explicitly disable default web-search, response-healing, context-compression, Fusion, and Pareto Router plugins. A successful response must report `strategy: "direct"` and no pipeline stages; missing strategy, unknown strategy, unknown pipeline stages, managed routing, plugin mutation, server tools, response healing, or context compression are treated as fixed-baseline failures rather than comparable evidence. Ordinary response-cache hits that omit router metadata and organization settings that prevent request-level plugin overrides are expected blockers for fixed-baseline public comparisons; use the private report and OpenRouter account/plugin settings to diagnose them. The private run provenance binds this request policy and public reports expose only a no-digest status for it.
- The no-spend case-set claim gate blocks exact duplicate and high-confidence lexical near-duplicate scored case content because duplicated or heavily templated evidence can inflate denominators, confidence intervals, and category evidence. Smoke-only duplicates are excluded because smoke cases are not scored.
- Difficulty metadata is a closed enum on case definitions and is included in case-set and manifest identity, but not in model-visible seed material or duplicate-content identity. Public cost-performance case-set gates require at least 30 scored cases in each difficulty bucket, at least five scored cases per difficulty bucket within each claim-eligible category, and at least five non-surface grader-evidence scored cases in each difficulty bucket; these are minimum coverage checks, not semantic difficulty calibration or proof of balance.
- Repetition is represented by `trialsPerCase`; duplicate configuration IDs are rejected because they corrupt paired metrics.
- `paired_vs_direct` point metrics are descriptive over available direct/target trial pairs. Pass-rate delta confidence intervals require exactly one non-smoke direct outcome and one non-smoke target outcome for every scored case-trial; a null interval means comparative interval evidence is unavailable even if partial paired point metrics exist.
- Comparable output-token caps.
- Persist model version, provider, price snapshot, prompt version, and usage.
- Persist sampling parameters, seed policy, and best-effort determinism metadata for every planned call.
- Hide model identity and price from the aggregator.
- Treat aggregator prompt/schema versions as evaluation conditions. `frugal-fusion-prompts-v2` requires compact comparison-ledger fields for coverage gaps and unique adopted claim IDs, rejects aggregator ledgers that cite non-blinded, stale, invented, duplicate, or contradictory claim IDs, and rejects consensus IDs that do not span at least two candidates. The ledger is still model-reported comparison metadata; the runtime checks ID consistency, not semantic completeness of every coverage or uniqueness judgment.
- Report pair-level fusion harm: direct passes while fusion fails on the same case and trial.
- Report unavailable paired metrics as `null`, not zero, when `direct` or the compared config is absent.
- Counterbalance configuration order by case and trial to reduce run-order effects, and report execution-position counts.
- Treat cases without deterministic graders as smoke tests only when explicitly marked.
- Use text graders for broad surface checks, choice graders for strict closed-label outputs, JSON path and schema-subset graders for strict structured outputs, numeric graders for finite numeric outputs with explicit tolerance or bounds, and citation-mechanics graders for bracketed source-id discipline.
- Treat choice, JSON, schema-subset, numeric, and citation-mechanics graders as authoring tools, not as sufficient evidence for broad reasoning quality or semantic source entailment; public claims still need category balance, holdout cases, at least a small non-surface grader-evidence cell overall, the same small non-surface cell in every difficulty bucket, and the same small non-surface cell in every claim-eligible category. Surface-text-only scored evidence remains useful during harness development but is blocked by public cost-performance claim gates.
- Derive grader evidence tiers from case definitions before model calls. The tiers are mechanical evidence-shape metadata, not semantic grading strength, task difficulty, or benchmark validity. Keep the tier classifier version separate from case-set fingerprints so changing classifier logic does not perturb execution seeds. Smoke-only cases are always classified as `smoke_only`; any configured smoke-only grader is validated for shape but counted separately as ignored evidence because runtime smoke grading does not use it.
- Keep grader diagnostics symbolic by default so expected values, extracted values, answer snippets, and JSON fragments do not bypass output-retention defaults.
- Exclude smoke-only cases from task pass rate, cost per pass, and fusion harm; report smoke completion separately.
- Include failed-attempt usage in total cost when a terminal error occurs after one or more calls have already completed.
- Validate paid failed-attempt usage with the same model, price, token, and budget checks used for successful calls.
- Treat configured seeds as reproducibility metadata, not as a guarantee of deterministic provider output.
- Treat run-provenance fingerprints and normalized CLI invocation metadata as private audit binding/context metadata, not as public proof. The resolved-config fingerprint is computed from allowlisted executed config fields after defaults are applied; unknown raw config keys, local paths, argv, and environment names are not included. The optional invocation metadata is self-reported, closed-enum `cli eval` context and does not replace the exact command or external run log. Run-provenance v2 binds resolved config, model-price snapshot, fixed OpenRouter request policy, and provider routing evidence as private-report metadata. The model-price fingerprint covers the sorted unique effective `PriceSnapshotEntry` records supplied for the evaluated run rather than the whole provider catalog.
- Include trial identity in seed material so repeated attempts can vary when providers honor seeds.
- Bootstrap confidence intervals resample cases as clusters, keeping all trials for a sampled case together.
- Report confidence intervals as unavailable when there is no scored bootstrap population.
- Baseline C uses the same model and same candidate prompt twice; fusion config validation requires two distinct candidate model IDs. Role-biased same-model panels are a separate future ablation.
- Runtime `auto` mode is a direct-only MVP placeholder and records closed-enum `autoRouting` metadata. It is not part of the fixed-mode evaluation matrix and must not be described as adaptive routing until offline trace evidence supports a separate router experiment.

## Deterministic Graders

- `json.requireValid` parses `answer.trim()` as strict JSON. Fenced or prose-wrapped JSON fails.
- `choice` graders normalize the whole answer and configured labels by trimming, collapsing whitespace, and lowercasing. They require the answer to be exactly one allowed label; prose, multiple labels, aliases, and substring matches fail. Prompts should instruct the model to return exactly one label. `choice_valid` checks the label is allowed, while `choice_expected` checks the expected label.
- JSON paths use dot segments plus array indexes, such as `steps[0].name`; wildcards and prototype-like paths are rejected before spend.
- JSON checks currently support required paths, primitive equality, string inclusion, array minimum length, and an internal schema subset. This is intentionally not a JSON Schema engine.
- `json.schemaSubset` validates the parsed answer against Frugal Fusion's internal schema subset. Supported schema keywords are `type`, object `properties`, object `required`, explicit object `additionalProperties`, array `items`, string `enum`, and numeric `minimum`/`maximum`. Unsupported keywords are rejected before spend. Object schemas must explicitly set `additionalProperties` so strictness is intentional. Schema-subset grader failures use symbolic diagnostics only.
- `number` extracts either the first finite number or one configured regex capture, then checks absolute tolerance and optional min/max bounds.
- Regex graders use a deliberately narrow regex subset. Text regexes reject grouping, alternation, and unbounded quantifiers; numeric extraction regexes allow exactly one simple numeric capture.
- Citation-mechanics graders parse bracket citations like `[S1]` against configured safe source IDs. They can require cited source IDs, a minimum number of unique cited sources, and literal claim snippets followed by the required source citation within a bounded window. Per required claim, authors can opt into immediate placement so the source citation must be the first non-whitespace text after the literal claim snippet. The citation grader config stores no source corpus text, but case prompts may still contain source snippets and remain governed by prompt-retention and public-report rules. Citation checks do not prove that a claim is entailed by a source.
- Grader evidence tier `structured_or_exact` means the case has closed-label choice, JSON path/schema-subset, numeric, citation-mechanics, exact, or normalized-exact deterministic checks only. `surface_text` means it uses keyword, regex, or length checks only. `mixed` means both families are configured. `smoke_only` cases are classified as smoke even when a grader object is present, because runtime scoring ignores the grader. `ungraded` is retained for defensive reporting and should be rejected for non-smoke cases.
